# 基于 SpringBoot 和 Vue 的本地特色美食论坛完整开发流程

## 1. 文档定位

本文档严格对照开题报告《基于 SpringBoot 和 Vue 的本地特色美食论坛的设计与实现》整理，面向毕业设计实施全过程，覆盖需求拆解、原型设计、UI 开发、功能开发、测试调试、论文配套、部署上线与答辩准备等完整环节。文档目标不是泛泛而谈，而是给出一套可直接执行、可分阶段交付、可用于毕业设计验收的实施方案。

## 2. 与开题报告对应的项目约束

### 2.1 课题名称

- 基于 SpringBoot 和 Vue 的本地特色美食论坛的设计与实现

### 2.2 研究目标

根据开题报告，系统的核心目标是：

- 面向本地美食爱好者、特色美食商家与平台管理员，搭建一个本地特色美食信息发布、交流与管理平台。
- 解决本地特色美食信息分散在短视频平台、社交媒体和综合点评网站中的问题。
- 提供内容发布、图片展示、分类检索、评论点赞、攻略展示、商家入驻、后台审核等功能。
- 通过前后端分离架构实现结构清晰、易维护、可扩展、运行稳定的 Web 系统。
- 在满足功能完整性的同时，保证交互体验优秀、界面美观统一，具备毕业设计展示与答辩价值。

### 2.3 报告中明确的功能模块

#### 用户端功能模块

- 用户注册与登录
- 用户信息管理
- 美食内容发布
- 美食信息浏览与查询
- 互动交流（评论、点赞）
- 美食攻略展示

#### 后台管理功能模块

- 用户管理
- 内容审核与管理
- 美食分类管理
- 商家入驻管理

### 2.4 报告中明确的技术栈

- 前端：`Vue`
- 后端：`Java` + `SpringBoot`
- 数据库：`MySQL`
- 接口风格：`RESTful API`
- 开发工具：`IntelliJ IDEA`、`VS Code`、`MySQL Workbench`、`Apifox`、`Git`

### 2.5 本项目的非功能目标

- 页面美观、布局合理、配色统一、字体规范
- 交互流畅，用户操作路径清晰
- 移动端与桌面端均可正常浏览
- 内容结构清晰，信息检索效率高
- 后台管理操作高效且风险可控
- 项目具备论文截图、系统演示、答辩展示条件

## 3. 总体实施原则

### 3.1 开发原则

- 严格围绕开题报告，不偏离研究对象与技术主线。
- 采用“先需求、后原型、再设计、再开发、先联调、后优化、最后部署”的顺序推进。
- 所有页面先出低保真原型，再出高保真 UI，再进入实际开发。
- 所有核心功能必须具备对应的数据表、接口文档、页面入口与测试用例。
- 所有 UI 页面必须同步输出视觉规范，不能边开发边随意堆页面。

### 3.2 UI 总原则

本项目的 UI 不应只满足“能用”，而应达到“适合毕设展示”的标准，因此从一开始就要把视觉设计作为主线任务，而不是最后补皮肤。

- 风格方向：现代轻量、生活化、偏内容社区风格，突出“美食、城市、分享”氛围。
- 色彩策略：主色建议使用暖橙、米白、深灰、食欲红棕的组合，强调温暖与食物质感。
- 字体策略：标题与正文区分层级，保证信息密度高但不压抑。
- 布局策略：首页、列表页、详情页、发布页、后台页风格统一，但功能区明确区分。
- 组件策略：按钮、输入框、卡片、标签、弹窗、分页、筛选条、评论区要统一规范。
- 响应式策略：至少适配 `1440px`、`1200px`、`768px`、`390px` 四档。
- 动效策略：以轻动画增强质感，如卡片悬浮、图片切换、标签过渡、提交反馈，不堆叠复杂动画。

## 4. 完整开发流程总览

建议将项目拆分为 10 个阶段推进：

1. 课题对齐与需求拆解
2. 业务建模与信息架构设计
3. 原型设计与 UI 视觉规范制定
4. 技术环境搭建与基础工程初始化
5. 数据库设计与接口规范设计
6. 前端 UI 页面开发
7. 后端业务功能开发
8. 联调、测试、调优与问题修复
9. 部署上线与演示环境准备
10. 论文截图、答辩材料与演示脚本整理

以下对每个阶段展开说明。

## 5. 第一阶段：课题对齐与需求拆解

### 5.1 目标

将开题报告中的研究目标、功能描述、角色边界、技术要求转化为可开发的任务清单，避免后期返工。

### 5.2 必做工作

#### 5.2.1 识别系统角色

- 普通用户：注册、登录、浏览、搜索、发布、点赞、评论、查看攻略、维护个人资料
- 商家用户：提交入驻信息、维护店铺展示资料、等待审核
- 管理员：用户管理、内容审核、分类维护、商家审核、平台内容管理

#### 5.2.2 拆解业务流程

需要画出以下核心流程：

- 用户注册登录流程
- 用户发布美食内容流程
- 用户浏览并搜索美食流程
- 用户评论与点赞流程
- 管理员审核内容流程
- 商家提交入驻并审核通过流程
- 美食攻略推荐与展示流程

#### 5.2.3 输出需求清单

将需求分为三类：

- 核心必做：登录注册、用户信息、美食发布、浏览查询、点赞评论、后台审核、分类管理、商家入驻
- 质量增强：攻略推荐、首页推荐区、热门榜单、标签筛选、图文详情优化
- 答辩加分：数据统计看板、内容热度排行、首页 Banner、卡片动画、后台审核状态流转

#### 5.2.4 形成需求文档

需求文档建议包含：

- 项目背景
- 角色说明
- 用例清单
- 页面清单
- 数据对象清单
- 功能优先级
- 非功能要求
- UI 质量要求

### 5.3 交付物

- `需求规格说明书`
- `角色与业务流程图`
- `功能清单与优先级表`

### 5.4 UI 在本阶段的落地要求

- 每个功能都要同步考虑页面入口与操作路径，不能只写“有这个功能”。
- 对每个页面补充“主视觉目标”，例如：首页强调氛围感，详情页强调内容沉浸，后台页强调效率与秩序。
- 在需求文档中明确 UI 验收项：统一色板、统一按钮样式、统一间距体系、统一栅格布局。

## 6. 第二阶段：业务建模与信息架构设计

### 6.1 目标

在正式画页面前，先把系统结构理清，确保内容组织合理、导航清晰、后续开发不混乱。

### 6.2 页面信息架构

建议前台页面结构如下：

- 首页
- 美食列表页
- 美食详情页
- 攻略列表页
- 攻略详情页
- 发布美食页
- 用户中心页
- 登录页
- 注册页
- 商家入驻申请页

后台页面结构建议如下：

- 管理后台首页
- 用户管理页
- 美食内容审核页
- 评论管理页
- 分类管理页
- 商家入驻审核页
- 攻略管理页
- 系统公告或轮播管理页

### 6.3 数据对象建模

至少需要明确以下实体：

- 用户 `user`
- 管理员 `admin`
- 美食分类 `food_category`
- 美食内容 `food_post`
- 美食图片 `food_image`
- 评论 `comment`
- 点赞记录 `like_record`
- 攻略 `strategy`
- 商家入驻信息 `merchant_application`
- 审核记录 `audit_record`

### 6.4 导航逻辑设计

- 前台顶部导航固定为：首页、美食广场、攻略推荐、商家入驻、个人中心
- 列表页采用“搜索栏 + 分类筛选 + 排序 + 卡片列表”
- 详情页采用“主内容 + 侧边推荐”结构
- 后台采用左侧菜单、右侧工作区布局

### 6.5 交付物

- `站点地图`
- `页面跳转关系图`
- `实体关系图 ER 图`
- `业务流程图`

### 6.6 UI 在本阶段的落地要求

- 先定义页面层级，再定义视觉层级，避免后期“哪里都想强调”。
- 首页、列表页、详情页、后台页分别定义不同的信息密度。
- 先确定卡片、标签、筛选区、评论区、审核表格在视觉上的主次关系。

## 7. 第三阶段：原型设计与 UI 视觉规范制定

### 7.1 目标

先把页面“做对”，再把页面“做美”。此阶段是毕设 UI 成败的关键。

### 7.2 原型设计流程

#### 7.2.1 低保真原型

先用 `Figma`、`Axure` 或 `墨刀` 画低保真，重点确认：

- 首页模块排序是否合理
- 列表筛选是否顺手
- 发布页表单是否完整
- 详情页内容是否有重点
- 后台审核操作是否高效

#### 7.2.2 高保真原型

在低保真确认后，再输出高保真 UI。建议页面优先级：

1. 首页
2. 美食列表页
3. 美食详情页
4. 登录注册页
5. 发布页
6. 用户中心页
7. 商家入驻页
8. 后台管理页

### 7.3 UI 视觉规范建议

#### 色彩规范

- 主色：暖橙或橙红，用于主按钮、重点标签、主视觉点缀
- 辅色：米白、浅奶油色，用于背景
- 中性色：深灰、浅灰，用于正文和分割线
- 状态色：成功绿、警告橙、错误红、审核蓝

#### 字体规范

- 页面主标题：`28px-36px`
- 模块标题：`20px-24px`
- 卡片标题：`16px-18px`
- 正文：`14px-16px`
- 辅助信息：`12px-14px`

#### 间距规范

- 页面左右边距：桌面端 `80px` 左右，平板端 `32px`，移动端 `16px`
- 模块上下间距：`24px / 32px / 48px`
- 卡片内边距：`16px / 20px / 24px`

#### 组件规范

- 按钮：主按钮、次按钮、文本按钮三种层级
- 卡片：统一圆角、阴影、悬浮状态
- 输入框：统一高度、边框、聚焦样式
- 标签：分类标签、状态标签、热度标签统一设计
- 空状态：无数据时必须有友好占位图与文案

### 7.4 核心页面 UI 设计要求

#### 首页

- 顶部 Banner 要有视觉记忆点，可使用城市美食图拼接或暖色渐变大图
- 首屏要快速传达“本地特色美食交流社区”的定位
- 下方按“热门推荐、分类入口、最新发布、优质攻略、入驻商家”布局
- 卡片图片比例统一，避免页面杂乱

#### 美食列表页

- 搜索框、分类筛选、地区标签、排序控件要置于首屏可见区域
- 列表卡片应包含封面、标题、分类、店铺、点赞数、发布时间
- 鼠标悬浮时卡片应有轻微抬升和阴影变化

#### 美食详情页

- 大图区域要突出视觉吸引力
- 右侧可放店铺信息、分类标签、互动区
- 评论区要清晰分层，支持时间、用户名、头像、内容区分
- 推荐内容区不宜喧宾夺主

#### 发布页

- 表单分组明确：基础信息、图片上传、内容描述、分类选择、店铺信息
- 上传区要有拖拽感和缩略图预览
- 表单错误提示要明确，不使用生硬报错

#### 用户中心页

- 展示头像、昵称、简介、我的发布、我的点赞、我的评论等信息
- 个人主页要有社区感，而不是后台感

#### 后台管理页

- 强调效率和秩序，采用浅背景、清晰表格、明确状态标签
- 审核操作按钮位置统一，避免误触
- 列表支持搜索、筛选、批量操作

### 7.5 交付物

- `低保真原型`
- `高保真视觉稿`
- `设计规范文档`
- `组件库样式清单`

## 8. 第四阶段：技术环境搭建与基础工程初始化

### 8.1 目标

搭建可运行、可协作、可扩展的前后端工程骨架。

### 8.2 后端初始化

- 使用 `SpringBoot` 创建项目
- 按照 `controller / service / serviceImpl / mapper / entity / dto / vo / config` 分层
- 接入数据库连接配置
- 统一返回体结构
- 统一异常处理
- 配置跨域
- 配置文件上传目录
- 完成基础登录鉴权方案

### 8.3 前端初始化

- 使用 `Vue` 初始化项目
- 建立 `views / components / api / router / stores / assets / styles` 目录
- 配置路由
- 配置状态管理
- 封装请求模块
- 建立全局主题变量和基础样式文件
- 预置通用组件：顶部导航、页脚、卡片、分页、弹窗、标签、上传组件

### 8.4 工具链配置

- `Git` 管理版本
- `Apifox` 管理接口文档
- `MySQL Workbench` 管理数据库
- `IDEA` 与 `VS Code` 分别负责后端和前端

### 8.5 UI 在本阶段的落地要求

- 前端初始化时即建立 `theme.css` 或 `variables.scss`
- 将颜色、圆角、阴影、字号、间距抽为变量，不要后期到处硬编码
- 提前定义栅格断点和响应式方案

### 9.5 UI 在本阶段的落地要求

- 列表页展示字段、详情页展示字段、卡片摘要字段要提前定义，避免后续接口反复改。

### 9.6 商家实名认证与资质体系扩展（新增）

为支撑“商家身份增强、内容认证、商家名片窗口、食品安全责任”需求，第五阶段新增如下数据设计要求：

#### 9.6.1 新增商家资料表 `merchant_profile`

- 设计目标：承接商家审核通过后的长期资料维护，不与 `merchant_application` 混用。
- 关键字段建议：
- `id`（PK）
- `merchant_user_id`（唯一，关联 `user.id`，用于识别商家账号）
- `merchant_name`、`store_name`、`store_address`、`store_phone`、`avg_price`
- `business_license_url`（营业执照）
- `food_safety_cert_url`（食品健康安全证）
- `status`（`PENDING_REVIEW` / `APPROVED` / `REJECTED` / `FROZEN`）
- `cert_version`（证件版本号，每次资质修改递增）
- `cert_updated_at`、`reviewed_at`、`review_remark`
- `created_at`、`updated_at`

#### 9.6.2 新增厨师信息表 `chef_info`

- 设计目标：结构化维护厨师团队背景，支撑“商家名片”展示。
- 关键字段建议：
- `id`（PK）
- `merchant_profile_id`（FK -> `merchant_profile.id`）
- `chef_name`、`title`（主厨/副厨/面点师等）、`years_of_exp`
- `specialty`（擅长菜系）、`certificates`（JSON 或文本）
- `avatar_url`、`intro`
- `sort_order`、`is_active`
- `created_at`、`updated_at`

#### 9.6.3 与帖子关联与认证判断

- 帖子作者仍通过 `food_post.author_user_id` 关联 `user.id`。
- 查询帖子时联查链路：
- `food_post.author_user_id` -> `merchant_profile.merchant_user_id`（且 `merchant_profile.status = APPROVED`）
- 满足条件则返回：
- `is_certified_merchant = true`
- `cert_label = 官方认证 / 某某商家直发`
- `merchant_profile_id`（用于详情页打开商家名片）

#### 9.6.4 过敏原责任模型（新增强制规则）

- 商家发布“官方菜品”时，菜品表必须记录过敏原标签。
- 建议新增或扩展：
- `food_dish`：`dish_name`、`ingredients`、`allergens`、`post_id`、`sort_order`
- 规则：商家角色发布帖子时，至少一条菜品；每条菜品必须有食材；过敏原至少做“已选择/无过敏原”显式声明。

#### 9.6.5 资质修改触发复审（强约束）

- 商家修改 `business_license_url` 或 `food_safety_cert_url` 后：
- 自动将 `merchant_profile.status` 置为 `PENDING_REVIEW`
- 该商家“官方认证标签”临时隐藏或降级为“认证复审中”
- 写入 `audit_record`，`audit_type = MERCHANT_PROFILE_CERT`
- 商家端仅允许提交变更，不允许自行恢复“已认证”状态

### 10.9 商家端差异化 UI 开发（新增）

#### 10.9.1 商家个人中心身份增强

当登录角色为商家（`role = MERCHANT`）时，个人中心应区别于普通用户：

- 新增“门店管理”模块：
- 门店基础信息（门店名、地址、电话、人均消费）
- 营业执照上传
- 食品健康安全证上传
- 证件状态与复审状态提示（通过/驳回/复审中）
- 新增“厨师管理”模块：
- 厨师列表、增删改、排序、擅长菜系、证书与简介

#### 10.9.2 商家帖子认证标签展示

- 在列表卡片和详情页标题区增加醒目标识：
- `官方认证`
- `某某商家直发`
- 视觉要求：高对比、轻金属感、避免俗艳，建立“可信赖”感。

#### 10.9.3 商家名片窗口（详情页交互）

- 点击“商家推荐”或商家头像/名称时，打开商家名片弹窗（Dialog/Drawer）：
- 左侧展示：营业执照、食品安全证缩略图（可预览）
- 右侧展示：厨师团队信息、热门菜品、门店基础信息
- 显示当前认证状态与最近审核时间

#### 10.9.4 状态流转前端约束

- 商家上传新证件后，界面即时提示“已提交复审，审核通过前将不显示认证标识”。
- 商家端不提供任何“恢复认证”操作入口，只能等待管理员审核。
