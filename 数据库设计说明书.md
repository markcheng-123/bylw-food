# 数据库设计说明书

项目名称：基于 SpringBoot 和 Vue 的本地特色美食论坛的设计与实现  
版本：v1.0  
日期：2026-03-18  

## 1. 设计目标与原则

### 1.1 设计目标

- 支撑开题报告中的用户端与后台管理模块实现。
- 支撑美食内容发布、分类检索、评论点赞互动、攻略展示、商家入驻与审核流程。
- 保证数据完整性、一致性与可维护性，同时兼顾常用查询的性能。

### 1.2 设计原则

- 使用 MySQL，建议字符集 `utf8mb4`，排序规则 `utf8mb4_general_ci` 或 `utf8mb4_0900_ai_ci`（依据 MySQL 版本）。
- 采用逻辑删除（`is_deleted`）与时间戳（`created_at`/`updated_at`）。
- 重要业务对象引入状态字段（例如内容审核状态）。
- 高频查询字段建立索引（分类、标题关键字、审核状态、时间）。
- 统一命名：表名使用下划线小写；主键 `id`；外键以 `_id` 结尾。

## 2. 概念模型（实体与关系）

### 2.1 核心实体（对照开题报告模块）

- 用户：发布、评论、点赞、维护资料
- 美食分类：用于规范化管理与查询
- 美食内容：论坛核心内容
- 美食图片：一条内容可对应多张图片
- 评论：对美食内容的互动
- 点赞记录：用户对内容的点赞去重与统计
- 攻略：美食攻略内容展示
- 商家入驻申请：商家资料提交与审核
- 审核记录：对内容/商家审核的过程记录

### 2.2 关系说明

- `food_post` 归属于一个 `food_category`
- `food_post` 由一个 `user` 发布
- `food_post` 可拥有多条 `food_image`
- `food_post` 可拥有多条 `comment`
- `user` 对 `food_post` 产生多条 `like_record`
- `merchant_application` 由一个 `user`（商家账号）提交，管理员审核
- `audit_record` 记录管理员对业务对象的审核动作

## 3. 状态枚举（建议）

### 3.1 内容审核状态（food_post.status）

- `DRAFT`：草稿（可选）
- `PENDING`：待审核
- `APPROVED`：已发布
- `REJECTED`：驳回
- `OFFLINE`：下架（可选）

### 3.2 商家入驻状态（merchant_application.status）

- `PENDING`：待审核
- `APPROVED`：通过
- `REJECTED`：驳回

## 4. 表结构设计（建议落库版本）

说明：字段类型与长度可在开发阶段结合实际组件与接口再细化，本说明书提供可直接实现的建议结构。

### 4.1 用户表：user

用途：存储普通用户与商家用户（通过 role/flag 区分）。

- id：BIGINT，PK，自增
- username：VARCHAR(64)，唯一，非空
- password_hash：VARCHAR(255)，非空
- nickname：VARCHAR(64)，非空
- avatar_url：VARCHAR(512)，可空
- phone：VARCHAR(32)，可空
- bio：VARCHAR(256)，可空
- role：VARCHAR(16)，非空（建议：`USER`/`MERCHANT`/`ADMIN`，也可将管理员独立表）
- status：TINYINT，非空（0 正常，1 禁用）
- created_at：DATETIME，非空
- updated_at：DATETIME，非空
- is_deleted：TINYINT，非空（0 否，1 是）

索引建议：

- uk_user_username(username)
- idx_user_role(role)
- idx_user_status(status)

### 4.2 美食分类表：food_category

- id：BIGINT，PK
- name：VARCHAR(64)，唯一，非空
- sort_order：INT，默认 0
- status：TINYINT（0 启用，1 停用）
- created_at / updated_at / is_deleted

索引建议：

- uk_category_name(name)
- idx_category_status(status)

### 4.3 美食内容表：food_post

用途：核心内容，支持发布、审核、检索。

- id：BIGINT，PK
- author_user_id：BIGINT，FK -> user.id
- category_id：BIGINT，FK -> food_category.id
- title：VARCHAR(128)，非空
- summary：VARCHAR(256)，可空（列表摘要，可从内容截断生成）
- content：TEXT，非空（图文内容的文本部分；如后续做富文本可存 HTML）
- cover_url：VARCHAR(512)，可空（封面）
- shop_name：VARCHAR(128)，可空
- shop_address：VARCHAR(256)，可空
- shop_phone：VARCHAR(32)，可空
- status：VARCHAR(16)，非空（见枚举）
- reject_reason：VARCHAR(256)，可空
- like_count：INT，非空，默认 0（冗余计数）
- comment_count：INT，非空，默认 0（冗余计数）
- view_count：INT，非空，默认 0（可选）
- published_at：DATETIME，可空（审核通过时写入）
- created_at / updated_at / is_deleted

索引建议：

- idx_post_category(category_id)
- idx_post_status(status)
- idx_post_created(created_at)
- idx_post_published(published_at)
- idx_post_author(author_user_id)
- FULLTEXT(title, content)（可选，取决于 MySQL 版本与检索方案）

### 4.4 美食图片表：food_image

- id：BIGINT，PK
- post_id：BIGINT，FK -> food_post.id
- image_url：VARCHAR(512)，非空
- sort_order：INT，默认 0
- created_at / updated_at / is_deleted

索引建议：

- idx_image_post(post_id)

### 4.5 评论表：comment

- id：BIGINT，PK
- post_id：BIGINT，FK -> food_post.id
- user_id：BIGINT，FK -> user.id
- content：VARCHAR(500) 或 TEXT，非空
- status：TINYINT（0 正常，1 屏蔽/删除）
- created_at / updated_at / is_deleted

索引建议：

- idx_comment_post(post_id)
- idx_comment_user(user_id)
- idx_comment_created(created_at)

### 4.6 点赞记录表：like_record

用途：实现“同一用户对同一内容不能重复点赞”，并支持取消点赞。

- id：BIGINT，PK
- post_id：BIGINT，FK -> food_post.id
- user_id：BIGINT，FK -> user.id
- status：TINYINT，非空（1 点赞，0 取消）
- created_at / updated_at / is_deleted

约束建议：

- unique(user_id, post_id)（保证去重）

索引建议：

- idx_like_post(post_id)
- idx_like_user(user_id)

### 4.7 攻略表：strategy

- id：BIGINT，PK
- author_user_id：BIGINT，可空（管理员或运营）
- title：VARCHAR(128)，非空
- cover_url：VARCHAR(512)，可空
- content：TEXT，非空
- status：VARCHAR(16)，非空（PENDING/APPROVED/OFFLINE 等，或简化为 ENABLED）
- created_at / updated_at / is_deleted

索引建议：

- idx_strategy_status(status)
- idx_strategy_created(created_at)

### 4.8 商家入驻申请表：merchant_application

- id：BIGINT，PK
- user_id：BIGINT，FK -> user.id（申请人账号）
- merchant_name：VARCHAR(128)，非空
- address：VARCHAR(256)，非空
- phone：VARCHAR(32)，非空
- intro：VARCHAR(512)，可空
- license_image_url：VARCHAR(512)，可空（材料）
- other_images_json：TEXT，可空（多图可用 JSON 存储，或拆表）
- status：VARCHAR(16)，非空（PENDING/APPROVED/REJECTED）
- reject_reason：VARCHAR(256)，可空
- created_at / updated_at / is_deleted

索引建议：

- idx_merchant_user(user_id)
- idx_merchant_status(status)
- idx_merchant_created(created_at)

### 4.9 审核记录表：audit_record

用途：记录管理员的审核动作，方便论文与系统管理展示。

- id：BIGINT，PK
- audit_type：VARCHAR(32)，非空（POST / MERCHANT / STRATEGY）
- target_id：BIGINT，非空（对应业务对象 id）
- operator_admin_id：BIGINT，FK -> user.id（管理员）
- action：VARCHAR(16)，非空（APPROVE/REJECT/OFFLINE）
- reason：VARCHAR(256)，可空
- created_at / updated_at / is_deleted

索引建议：

- idx_audit_type_target(audit_type, target_id)
- idx_audit_operator(operator_admin_id)
- idx_audit_created(created_at)

### 4.10 商家资料表：merchant_profile（新增）

用途：存储“审核通过后的商家正式资料”，支撑帖子认证标签与商家名片展示。

- id：BIGINT，PK
- merchant_user_id：BIGINT，唯一，FK -> user.id
- merchant_name：VARCHAR(128)，非空
- store_name：VARCHAR(128)，非空
- store_address：VARCHAR(256)，非空
- store_phone：VARCHAR(32)，可空
- avg_price：INT，可空
- business_license_url：VARCHAR(512)，非空
- food_safety_cert_url：VARCHAR(512)，非空
- intro：VARCHAR(512)，可空
- status：VARCHAR(24)，非空（`PENDING_REVIEW`/`APPROVED`/`REJECTED`/`FROZEN`）
- cert_version：INT，非空，默认 1
- reviewed_at：DATETIME，可空
- review_remark：VARCHAR(256)，可空
- created_at / updated_at / is_deleted

索引建议：

- uk_merchant_profile_user(merchant_user_id)
- idx_merchant_profile_status(status)
- idx_merchant_profile_store(store_name)

### 4.11 厨师信息表：chef_info（新增）

用途：维护厨师团队资料，供商家名片和详情页展示。

- id：BIGINT，PK
- merchant_profile_id：BIGINT，FK -> merchant_profile.id
- chef_name：VARCHAR(64)，非空
- title：VARCHAR(64)，可空
- years_of_exp：INT，可空
- specialty：VARCHAR(128)，可空
- certificates：VARCHAR(512)，可空（或 JSON）
- avatar_url：VARCHAR(512)，可空
- intro：VARCHAR(512)，可空
- sort_order：INT，默认 0
- is_active：TINYINT，默认 1
- created_at / updated_at / is_deleted

索引建议：

- idx_chef_profile(merchant_profile_id)
- idx_chef_sort(merchant_profile_id, sort_order)

## 5. 初始化数据建议

- 分类初始化：至少 8-12 个分类（如：小吃、面食、烧烤、甜品、饮品、川湘、粤菜、火锅、早餐、夜宵等）
- 管理员账号：初始化 1 个管理员，密码首次登录后修改（论文中可说明）。
- 演示数据：准备 12+ 条美食内容、4+ 条攻略、若干评论点赞数据。

## 6. 数据一致性与并发说明

- like_count/comment_count 建议作为冗余字段，用于列表与详情快速展示。
- 点赞的最终一致性策略：写 like_record 后增减 like_count（需处理并发，推荐数据库层唯一约束 + 事务）。
- 评论数同理：新增评论后增 comment_count；删除/屏蔽评论后同步调整（或定时校准）。

### 6.1 商家认证一致性规则（新增）

- 认证标签展示条件：`food_post.author_user_id` 能关联到 `merchant_profile.merchant_user_id` 且 `merchant_profile.status = APPROVED`。
- 商家修改执照或食品安全证时，必须自动触发复审：
- `merchant_profile.status` -> `PENDING_REVIEW`
- `cert_version` + 1
- 写入 `audit_record`（建议 `audit_type = MERCHANT_PROFILE_CERT`）
- 复审期间帖子不展示“官方认证/商家直发”标签，防止用户误判。
